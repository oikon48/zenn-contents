# Oikonの記事の文体特徴

このドキュメントは、Oikonの技術記事の文体と特徴をまとめたものです。

## 基本的な文体

### 1. カジュアルで親しみやすい口調

- 技術記事でありながら、読者に話しかけるような口語調
- 硬すぎず、かといって砕けすぎない絶妙なバランス
- 例：「Oikonです。普段はAIツール、特にClaude Codeで遊んでいます。」

### 2. 実体験ベースの導入

- 自己紹介から始まる
- 実際に体験したこと、試したことをベースに語る
- 登壇、ハッカソン、実際の運用など具体的な背景を共有
- 例：「先日、Claude Code Meetup Tokyoがあり登壇の機会をいただきました。」

### 3. 謙虚な姿勢

- 自分の経験の浅さを認める
- 読者へのリスペクトを忘れない
- 例：「モバイルアプリ開発者の方々には既知の情報も多いと思いますが、筆者のモバイルアプリ開発歴は1ヶ月未満ということで、ご容赦いただければ幸いです。」

## 構成の特徴

### 1. 明確な構造化

- セクションは`##`で明確に区切る
- サブセクションは`###`を使用
- 読者が目次から目的の情報にアクセスしやすい

### 2. 視覚的な補助

- スクリーンショットや図を積極的に活用
- 画像には適切な代替テキストとパス指定
- 例：`![iPhone単体での開発ワークフロー](/images/dev-from-mobile/iphone-workflow.png)`
- サイズ指定も活用：`![chatgpt-code](/images/dev-from-mobile/chatgpt-code.png =400x)`

### 3. 箇条書きの多用

- 複数の項目は箇条書きで整理
- 読みやすく、スキャンしやすい
- ステップバイステップの説明に効果的

## 説明スタイル

### 1. 全体像の提示

- まず全体のワークフローや概要を図で示す
- その後、各ステップを詳しく説明
- 例：「実際に運用しているワークフローです。以下の5つのステップで構成されています」

### 2. 具体的なコマンドとコード例

- 実際に使えるコマンドを明示
- コードブロックで見やすく提示
- 環境依存の部分は注釈を添える
- 例：
```bash
# "/path/to/vault" はご自身の環境に合わせて修正してください
claude mcp add obsidian-mcp-tools -s user -- "/path/to/vault/.obsidian/plugins/mcp-tools/bin/mcp-server"
```

### 3. 比較と選択肢の提示

- 複数の方法がある場合は比較して提示
- 個人的な感想も含めて選択の指針を示す
- 例：「両方使ってみた個人的な感想ですが、Codex Cloudの方が実装内容が良いことが多かったです」

### 4. 注意点とTips

- Zennの記法（`:::message`など）を活用
- 重要な注意点は目立たせる
- Optional（任意）な要素は明示

## 文章の細かい特徴

### 1. 強調表現

- 重要な部分は**太字**で強調
- 特に読者に注意してほしい点を明確に

### 2. 一人称の使用

- 「筆者」「自分」「個人的には」など
- 客観と主観を使い分ける

### 3. 補足説明

- カッコ書きで補足を追加
- 例：「ClaudeとCodexは片方だけでも実現できます。個人的には複数のAIエージェントに相互フィードバックさせるのが好きなので、複数のAIサービスを採用しています。」

### 4. リンクの自然な埋め込み

- 公式ドキュメントや参考記事は本文中に埋め込む
- スピーカーデッキ、ツイート、外部記事を積極的に参照

## 記事の締めくくり

### 1. まとめセクション

- 記事の要点を簡潔に再掲
- ワークフローや図を再掲示することも
- 読者が次に何をすべきか明確に

### 2. 宣伝・フォロー誘導

- Xのフォローリンクを自然に
- 関連イベントの告知
- 謙虚かつ親しみやすい表現
- 例：「Xでも情報発信しているので、フォローしていただけると励みになります！」

### 3. 参考文献

- 記事で引用した全てのリンクを列挙
- 公式ドキュメント、外部記事、自分の過去記事など

## 記事のメタ情報（Zennフロントマター）

```yaml
topics: [claudecode,ai,tools,実際のトピック名]
published: true
published_at: "2025-10-18 18:06"
---
```

## 文体のDo's and Don'ts

### Do's（すべきこと）

- ✅ 実体験を基に語る
- ✅ 具体的なコマンドやコード例を示す
- ✅ 画像で視覚的に補助
- ✅ ステップバイステップで説明
- ✅ 比較と個人的な感想を添える
- ✅ 謙虚な姿勢を保つ
- ✅ 読者が次のアクションを取れるようにする

### Don'ts（避けるべきこと）

- ❌ 上から目線の説明
- ❌ 抽象的すぎる説明
- ❌ 環境情報の欠如
- ❌ 説明なしのコードの羅列
- ❌ 実際に試していない内容
- ❌ 参考文献の欠如
